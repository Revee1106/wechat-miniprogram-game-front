# 微信小游戏 Canvas 前端迁移设计

日期：2026-04-07

项目：`wechat-miniprogram-game-front`

## 目标

将当前基于微信小程序页面体系的前端，迁移为基于微信小游戏原生 Canvas 2D 的单场景前端，并同时重做 UI 结构。

迁移后的首期目标不是保留原有 6 个页面，而是形成：

- 单主场景常驻
- 以抽屉和弹层承载次级玩法
- 主循环、资源动作、洞府、炼丹都可直接操作
- 以微信小游戏为唯一前端目标

## 已确认约束

- 渲染方案：纯原生微信小游戏 Canvas 2D
- 信息架构：单主场景 + 抽屉 / 弹层
- 首期范围：经营优先，不接受只留入口
- 代码策略：直接替换，小游戏作为唯一前端目标

## 当前问题

当前仓库虽然已经补了 `game.js` 和 `game.json`，能在小游戏模式下启动逻辑，但真正的 UI 仍然依赖：

- `app.json`
- `pages/**`
- `components/**`
- `*.wxml`
- `*.wxss`

这些都属于小程序页面体系，小游戏模式不会渲染，因此开发者工具里出现“逻辑已运行但右侧黑屏”的现象。

根因不是接口层，而是渲染层完全不匹配小游戏运行模型。

## 迁移原则

### 1. 保留现有业务接入层

尽量保留现有：

- `utils/api.js`
- `utils/run-store.js`
- `utils/breakthrough.js`
- 与后端 `/api/**` 的协议

原因：

- 当前后端主流程已经稳定联通
- 现有 store 已经封装了推进时间、事件结算、突破、洞府、炼丹、转生等动作
- UI 重写已经足够重，不应该同时重写业务接入层

### 2. 彻底替换页面层

以下结构不再作为主前端实现继续扩展：

- `pages/**`
- `components/**`
- `app.json`
- 所有 `wxml/wxss` 页面渲染

它们可以在迁移过程中短暂保留用于比对，但不再是最终运行路径。

### 3. 用轻量 Canvas UI 系统代替页面组件

不引入重型引擎，改为在仓库内构建一套只服务当前项目的轻量 UI 层，支持：

- 文本绘制
- 面板绘制
- 按钮点击
- 列表滚动
- 抽屉展开收起
- 弹层遮罩
- 命中区域管理

### 4. 优先保证功能闭环，再做视觉细化

首期重点是让小游戏版真正可操作：

- 开局
- 推进时间
- 选事件
- 资源动作
- 突破
- 洞府建造升级
- 炼丹和服用
- 死亡转生

视觉重做会同步进行，但不以炫技动画为优先级。

## 新信息架构

## 主舞台

小游戏启动后进入单一主舞台，不再跳转多个页面。

主舞台承担：

- 角色核心状态摘要
- 当前阶段 / 当前轮次 / 当前寿元
- 当前主叙事区
- 主按钮区
- 快速入口区

主舞台上的主操作永远只有一个优先焦点：

- 没有待处理事件时：`推进时间`
- 有待处理事件时：打开事件弹层并优先完成决策
- 角色死亡时：进入结算弹层并阻止继续推进

## 抽屉与弹层结构

### 1. 事件弹层

用途：

- 展示当前事件标题和正文
- 展示可选选项
- 展示选项是否可选及禁用原因
- 在结算后展示结果摘要

这是最优先的主决策弹层。

### 2. 背包 / 资源抽屉

用途：

- 展示统一资源
- 选择出售资源
- 灵石转修为
- 展示操作后的即时收益反馈

### 3. 修行抽屉

用途：

- 展示当前境界与突破目标
- 展示突破条件
- 发起突破

### 4. 洞府抽屉

用途：

- 展示设施卡片
- 展示设施等级、产出、维护和下一级成本
- 发起建造和升级
- 展示停摆风险与最近结算摘要

### 5. 炼丹抽屉

用途：

- 展示丹方列表
- 展示当前炉次
- 发起普通开炉 / 借灵泉开炉
- 展示库存并支持服用

### 6. 结算弹层

用途：

- 展示死亡后的结果摘要
- 展示轮回相关信息
- 发起转生

## 视觉方向

延续当前“纸墨 / 山海 / 静气”的整体气质，但改成更适合小游戏的结构。

具体原则：

- 主舞台只保留一块强视觉中心区域
- 不再用长页面堆叠卡片
- 信息密度集中在抽屉内部
- 常驻区强调状态感，抽屉区强调可操作性
- 结果反馈采用轻弹层或短驻留 toast，而不是页面跳转

## 目录建议

在仓库内新增小游戏前端核心目录，例如：

```text
src/game/
  adapters/
  core/
  screens/
  ui/
  view-models/
  theme/
```

建议职责如下：

- `adapters/`
  - 对接 `utils/api.js`
  - 对接 `utils/run-store.js`
  - 统一把 store 结果转为渲染层输入

- `core/`
  - 游戏启动
  - Canvas 初始化
  - 尺寸适配
  - 帧循环
  - 输入事件分发
  - UI 层级管理

- `screens/`
  - 主舞台
  - 事件弹层
  - 资源抽屉
  - 修行抽屉
  - 洞府抽屉
  - 炼丹抽屉
  - 结算弹层

- `ui/`
  - Button
  - Panel
  - Drawer
  - Modal
  - ScrollList
  - TextBlock
  - Badge
  - Toast

- `view-models/`
  - 把 run state 整理成适合渲染层直接消费的结构

- `theme/`
  - 颜色、字号、间距、圆角、阴影、排版规范

## 渲染模型

采用原生 Canvas 2D。

首期不做复杂场景树框架，但要有以下基础能力：

- 根容器渲染
- 元素按层级绘制
- 命中区域注册
- 遮罩层阻断点击
- 抽屉动画进度
- 可滚动区域偏移

建议采用“状态驱动重绘”而不是持续高频复杂动画：

- 状态变化时请求重绘
- 抽屉开关或滑动时按帧重绘
- 静止态尽量降低无意义绘制

## 输入模型

统一基于小游戏触摸事件实现：

- `wx.onTouchStart`
- `wx.onTouchMove`
- `wx.onTouchEnd`

必须支持：

- 点击按钮
- 点击空白关闭弹层
- 抽屉区域滚动
- 拖动抽屉关闭
- 阻止穿透点击

交互判定采用命中区域表：

- 每次渲染后更新可点击元素矩形
- 触摸结束时按 z-index 倒序命中

## 状态组织

## 数据来源

主要仍来自 `utils/run-store.js`。

Canvas 层不直接管理业务真相状态，只持有：

- 当前 UI 状态
- 当前打开的抽屉 / 弹层
- 当前选择中的资源或配方
- 当前滚动偏移
- Toast 和确认框状态

## 推荐分层

### 业务层

- run store
- API 调用

### 视图模型层

负责把原始 run state 变成：

- 顶部摘要
- 主舞台卡片内容
- 事件选项列表
- 设施卡片列表
- 丹方卡片列表
- 背包动作列表

### 渲染层

只接收视图模型和 UI 状态进行绘制。

## 迁移顺序

## Phase 1：小游戏基础框架

交付物：

- 单一 Canvas 启动入口
- 分辨率适配
- 基础绘制器
- 基础按钮 / 面板 / 弹层 / 抽屉
- 输入分发与命中系统

目标：

- 在开发者工具小游戏模式下能稳定看到主舞台，不再黑屏

## Phase 2：主循环舞台

交付物：

- 主舞台状态摘要
- 推进时间按钮
- 事件触发后自动打开事件弹层
- 事件结算结果展示
- 资源抽屉
- 修行抽屉

目标：

- 开局、推进时间、选事件、出售资源、灵石转修为、突破可完整跑通

## Phase 3：洞府抽屉

交付物：

- 洞府设施卡片
- 建造和升级操作
- 最近月结摘要
- 停摆风险确认

目标：

- 洞府从“页面”迁移为“抽屉内可操作系统”

## Phase 4：炼丹抽屉

交付物：

- 丹方列表
- 当前炼制状态
- 普通开炉 / 借灵泉开炉
- 库存与服用

目标：

- 炼丹在小游戏内完成闭环，不再依赖页面跳转

## Phase 5：结算与旧结构收尾

交付物：

- 结算弹层
- 转生流程
- 清理旧页面依赖
- 收缩或删除不再使用的小程序页面测试

目标：

- 小游戏成为唯一有效前端入口

## 手动验收

在微信开发者工具小游戏模式中确认：

- 启动后可见主舞台
- 点击推进时间可触发事件
- 事件弹层可完成选择
- 资源动作可执行
- 洞府建造 / 升级可执行
- 炼丹 / 服用可执行
- 死亡后进入结算弹层
- 转生后可回到新一局

## 风险与注意事项

### 1. `run-store.js` 已混入部分页面导向逻辑

需要逐步把页面专属拼装逻辑抽到 view model 层，否则 Canvas 层会继续堆业务格式化代码。

### 2. 中文排版与文本裁切

Canvas 文字绘制需要自己处理：

- 折行
- 最大宽度
- 行高
- 截断

这是小游戏 UI 体验的关键风险。

### 3. 滚动与点击穿透

抽屉、列表、遮罩重叠较多，必须从第一阶段就做好命中和层级规则。

### 4. 合法域名与本地联调

小游戏真机和部分环境下会受 HTTPS / TLS / 合法域名限制，开发期需明确：

- 仅开发者工具本地调试
- 或准备可用的 HTTPS 调试域名

### 5. 旧页面体系清理节奏

直接替换不代表一步删光，迁移期间允许旧页面暂存作为参考，但不能继续把新功能写回 `pages/**`。

## 验收标准

- 小游戏模式下进入项目后不再黑屏
- 无需依赖控制台命令即可完成核心流程操作
- 主循环、资源动作、洞府、炼丹都能在 Canvas UI 中直接完成
- 旧小程序页面不再是主运行路径
- 现有后端接口与业务协议不需要为迁移而重写

## 结论

本次迁移本质上是：

`保留业务接入层，彻底替换页面渲染层，建立适配微信小游戏的 Canvas 单场景前端。`

这是当前成本、风险和交付节奏之间最合理的方案。
